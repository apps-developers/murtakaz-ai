# UC-05 — المسؤول يُعدّ حساب مستخدم جديد

<div dir="rtl">

## ملخص حالة الاستخدام

| الحقل | القيمة |
|-------|--------|
| **المعرّف** | UC-05 |
| **العنوان** | المسؤول ينشئ حساب مستخدم جديد ويكلّفه بكيانات مؤشرات الأداء |
| **الجهة الفاعلة الرئيسية** | مسؤول المؤسسة (ADMIN) |
| **الجهات الفاعلة الثانوية** | المستخدم الجديد (مدير)، النظام |
| **المحفّز** | انضمام موظف جديد إلى المؤسسة ويحتاج للوصول إلى مؤشرات أداء محددة |
| **الشروط المسبقة** | المسؤول مسجّل دخوله؛ توجد أنواع كيانات وكيانات مُهيَّأة في المؤسسة |
| **الشروط اللاحقة** | المستخدم الجديد يمكنه تسجيل الدخول؛ يرى الكيانات المكلَّف بها؛ يمكنه إدخال القيم وإرسالها |
| **الأولوية** | متوسطة |

## رحلة المستخدم

```
يسجّل المسؤول دخوله
      │
      ▼
ينتقل إلى الإدارة ← المستخدمون
      │
      ▼
تُحمَّل صفحة المستخدمين
      │  يراجع المستخدمين الحاليين
      ▼
ينقر "+ مستخدم جديد"
      │
      ▼
يفتح حوار إنشاء المستخدم
      │
      ▼
يملأ بيانات المستخدم
      │  الاسم، البريد الإلكتروني، كلمة المرور، الدور، المسمى الوظيفي، المدير المباشر
      ▼
ينقر "حفظ"
      │
      ▼
أُنشئ المستخدم — يظهر في القائمة
      │
      ▼
ينتقل المسؤول إلى الكيانات المراد التكليف بها
      │
      ▼
يفتح تفاصيل الكيان ← تبويب التكليفات
      │
      ▼
ينقر "+ تكليف مستخدم" ← يختار المستخدم الجديد
      │
      ▼
يُحفظ التكليف
      │
      ▼
يُشارك المسؤول بيانات الاعتماد مع المستخدم الجديد
      │
      ▼
يسجّل المستخدم الجديد دخوله ← يرى مؤشرات الأداء المكلَّف بها
      │
      ▼
اكتملت الرحلة
```

---

## التدفق الرئيسي (خطوةً بخطوة)

1. يسجّل المسؤول دخوله وينتقل إلى **الإدارة** في الشريط الجانبي.
2. في لوحة الإدارة (`/<locale>/admin`)، ينقر المسؤول على **المستخدمون** (أو ينتقل مباشرةً إلى `/<locale>/admin/users`).
3. تُحمَّل صفحة قائمة المستخدمين وتعرض جميع المستخدمين الحاليين مع اسم كل منهم وبريده الإلكتروني ودوره ومديره وتاريخ إنشائه.
4. ينقر المسؤول على **+ مستخدم جديد**.
5. يفتح حوار **إنشاء مستخدم** بالحقول التالية:
   - **الاسم** (مطلوب) — الاسم الكامل، مثل: "أحمد العتيبي"
   - **البريد الإلكتروني** (مطلوب) — يجب أن يكون فريداً داخل المؤسسة
   - **كلمة المرور** (مطلوبة) — كلمة المرور الأولية، تُشارَك بشكل آمن مع المستخدم
   - **الدور** (مطلوب) — `ADMIN` أو `EXECUTIVE` أو `MANAGER`
   - **المسمى الوظيفي** (اختياري) — مثل: "مدير الأداء"
   - **المدير المباشر** (اختياري) — يربطه بمديره في الهيكل التنظيمي
6. يملأ المسؤول جميع الحقول المطلوبة وينقر على **حفظ**.
7. يُنشئ النظام المستخدم الذي يظهر فوراً في قائمة المستخدمين ويمكنه تسجيل الدخول على الفور.
8. لمنح المستخدم الوصول إلى مؤشرات أداء محددة، ينتقل المسؤول إلى كل كيان معني:
   - يفتح صفحة قائمة الكيانات (`/<locale>/entities/<typeCode>`)
   - ينقر على اسم الكيان لفتح صفحة التفاصيل
9. في صفحة تفاصيل الكيان، ينقر المسؤول على تبويب **التكليفات**.
10. ينقر المسؤول على **+ تكليف مستخدم**، يبحث عن اسم المستخدم الجديد ويختاره.
11. يُحفظ التكليف. أصبح بإمكان المستخدم الجديد الآن:
    - رؤية هذا الكيان في قائمة الكيانات الخاصة به
    - إدخال القيم وإرسالها لهذا الكيان
    - رؤيته في لوحة متابعة المدير الخاصة به
12. يُشارك المسؤول بيانات الاعتماد بشكل آمن مع المستخدم الجديد (بريد إلكتروني، محادثة، إلخ).
13. يسجّل المستخدم الجديد دخوله، ينتقل إلى صفحة النظرة العامة، ويرى مؤشرات الأداء المكلَّف بها في قسم "يحتاج انتباهاً" وقائمة الكيانات.

## التدفقات البديلة

### البديل أ — البريد الإلكتروني موجود بالفعل
في الخطوة 5، البريد الإلكتروني موجود مسبقاً في النظام.
- يُظهر النظام خطأ تحقق: "البريد الإلكتروني مستخدم بالفعل."
- يتحقق المسؤول مما إذا كان المستخدم لديه حساب بالفعل ويستخدم وظيفة تعديل المستخدم لتحديث دوره بدلاً من ذلك.

### البديل ب — تكليف المستخدم بكيانات كثيرة
في الخطوة 8، يوجد عدد كبير من الكيانات للتكليف بها.
- يستخدم المسؤول صفحة **المسؤوليات** (`/<locale>/responsibilities`) التي توفر واجهة مركزية لإدارة جميع تكليفات المستخدمين بالكيانات.

### البديل ج — المستخدم الجديد يحتاج وصول "تنفيذي" (للقراءة فقط)
في الخطوة 5، يضبط المسؤول الدور = `EXECUTIVE`.
- لا حاجة لتكليفات بالكيانات — التنفيذيون يرون جميع الكيانات تلقائياً.
- يتخطى المسؤول الخطوات 8–11.

### البديل د — تغيير دور مستخدم لاحقاً
بعد الإعداد، تتغير مسؤوليات المستخدم.
- ينتقل المسؤول إلى الإدارة ← المستخدمون، ينقر اسم المستخدم، ويستخدم أيقونة التعديل لتحديث حقل الدور.
- يسري الدور الجديد عند تحديث المستخدم للصفحة.

## القواعد التجارية

- يجب أن تكون عناوين البريد الإلكتروني فريدة داخل المؤسسة — لا يمكن لمستخدمَين مشاركة نفس البريد الإلكتروني.
- تُخزَّن كلمات المرور على شكل تجزئات bcrypt — لا يرى المسؤول كلمة المرور بعد إنشائها.
- المستخدم بدور `MANAGER` يرى **فقط** الكيانات المكلَّف بها (عبر `UserEntityAssignment`) أو التي هو مالكها (`ownerUserId`).
- المستخدم بدور `EXECUTIVE` يرى جميع الكيانات تلقائياً — لا حاجة لتكليفات.
- حذف المستخدم هو **حذف ناعم** — تُحفظ بياناته التاريخية (إدخالات القيم، الاعتمادات) لأغراض التدقيق.
